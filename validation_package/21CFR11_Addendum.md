## 1. Purpose

This addendum provides a field-by-field mapping of the `Part11AuditTrail` JSON
output to the applicable requirements of **21 CFR Part 11 — Electronic Records;
Electronic Signatures** (FDA, 1997).

It demonstrates that every execution of the `BMR-Date-Sequence-Validator` produces
an electronic record that satisfies the data integrity requirements of Subpart B
(§ 11.10) without requiring any additional manual steps.

## 2. Regulatory Mapping Table

| JSON Output Field | 21 CFR Part 11 Reference | Requirement Text (excerpt) | How This Agent Satisfies It |
|---|---|---|---|
| `execution_id` (UUID v4) | § 11.10(d) | "…limit system access to authorized individuals…" | Each execution is uniquely identified and non-repudiable |
| `timestamp_utc` (ISO 8601 UTC) | § 11.10(e) | "…use of secure, computer-generated, time-stamped audit trails…" | Timestamp is generated by the system clock at execution time; operator cannot alter it |
| `input_sha256` (SHA-256 hex) | § 11.70 | "Electronic signatures and handwritten signatures executed to electronic records shall be linked to their respective electronic records to ensure that the signatures cannot be excised, copied, or otherwise transferred…" | SHA-256 binds the audit trail irreversibly to the exact input text. Any alteration of the input changes the hash, making tampering detectable |
| `agent_name` + `agent_version` | § 11.10(a) | "…systems validation to ensure accuracy, reliability, consistent intended performance, and the ability to discern invalid or altered records…" | Agent name and semantic version are embedded in every record |
| `schema_version` | § 11.10(a) | Same as above | Schema versioning ensures forward compatibility and traceability of the validation state |
| `status` (PASS / FAIL) | § 11.10(e) | "…audit trails…to capture the date and time of operator entries and actions that create, modify, or delete electronic records…" | Deterministic PASS/FAIL prevents ambiguous outcomes |
| `rules_evaluated[].rule_id` | § 11.10(e) | Same as above | Each rule application is logged with a unique, human-readable identifier |
| `rules_evaluated[].result` | § 11.10(e) | Same as above | Result is one of three controlled vocabulary values: PASS, FAIL, ERROR |
| `rules_evaluated[].evidence.location` | § 11.10(e) | "…human readable form…" | Line number from the source document is preserved for manual traceability |
| `rules_evaluated[].evidence.found_value` | § 11.10(e) | Same as above | Raw extracted string from the batch record — no transformation |
| `rules_evaluated[].evidence.expected_value` | § 11.10(e) | Same as above | Expected value or format as defined in the SOP/specification |
| `rules_evaluated[].evidence.delta` | § 11.10(e) | Same as above | Human-readable deviation description for the deviation log |
| `summary.format_failures` + `summary.sequence_failures` | § 11.10(e) | Same as above | Aggregate counts for CAPA and trending dashboards |
| `execution_metadata.runtime_ms` | § 11.10(a) | "…consistent intended performance…" | Execution time documented for performance qualification |
| `execution_metadata.peak_ram_mb` | § 11.10(a) | Same as above | Memory footprint documented — demonstrates fitness for constrained GxP infrastructure |

## 4. Audit Trail Storage Recommendations

The `Part11AuditTrail` JSON output should be stored in a **21 CFR Part 11 compliant
document management system** (e.g., Veeva Vault, OpenText Documentum, or an FDA-validated
EDMS) with the following metadata applied:

| Metadata Field | Recommended Value |
|---|---|
| Document Type | `QA-Audit-Trail-Electronic` |
| Document ID | Value of `execution_id` field |
| Effective Date | Value of `timestamp_utc` field |
| Status | Value of `status` field |
| Source Hash | Value of `input_sha256` field |

---

## 5. Electronic Signature Note

This agent produces **electronic records** (21 CFR Part 11 Subpart B) but does **not**
create electronic signatures (Subpart C). If a human reviewer must countersign the
audit trail, the handwritten or electronic signature should reference the
`execution_id` UUID to create an unambiguous link between the record and the reviewer.
